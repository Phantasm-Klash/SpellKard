# Client Architecture

SpellKard owns input, presentation, local practice, replay playback, UI, and asset/theme handling.

In online play, the client is not authoritative. It sends input and card requests to Gensoulkyo and presents server-approved state.

The local prototype may run gameplay logic directly so movement, hitbox, graze, bomb, and replay feel can be tuned quickly.

## Local STG Loop

The current Godot prototype uses a fixed local tick for gameplay simulation and renders from the latest local state. This keeps the prototype close to the replay and online model, where authoritative state is generated from tick-indexed input and pattern data.

Bullet generation is split into two layers:

- `bullet_math.gd`: deterministic math helpers, angle helpers, polar/ellipse/wave/flower helpers, seeded random numbers, and card-style modifier application.
- `bullet_pattern_library.gd`: reusable pattern emitters for ring, safe-gap ring, n-way, seeded random arc, edge-spawn lanes, split chain, delayed blossom, telegraphed burst, charge-convert burst, limited homing, flower weave, sine stream, snake stream, curtain wall, aimed burst, vortex force fields, laser-warning curtain, cross warning lasers, extending capsule lasers, reflected polyline lasers, dynamic wave polyline lasers, orbital lattice bullets, orbit-hold release bullets, two-phase velocity-shift bullets, radius pulse/growing-orb bullets, origin-relative path-following bullets, Bezier path-following bullets, and moving trail emitters.
- `boss_pattern_catalog.gd`: auditable Boss pattern taxonomy and executable coverage gates for common Boss bullet families: aimed shots, n-way fans, radial rings, spirals, warning/persistent lasers, curve lanes/paths, delayed speed changes, split/transform carriers, deterministic seeded random patterns, and multi-phase Boss scripts. The standalone catalog check now verifies each catalog type emits deterministic bullets, seeded families change with seed, shaped lasers carry line/polyline metadata and continuous-graze cooldowns, and delayed/carrier behaviors can spawn child bullets through the shared bullet engine.
- `stage_select_model.gd`: local practice stage selection with four stage configs, difficulty/tempo/recommended-character metadata, localized stage/pattern labels, UI-ready rows for the current stage and its pattern set, and stage briefing rows that summarize the selected stage's math-route, peak density/danger, readability hints, and recommended self mode.
- `pattern_lab_model.gd`: read-only analysis rows for the current stage's patterns. It exposes the math basis, math family, key parameter summary, spawn-rate estimate, density band, danger band, and readability hint for each pattern so the practice/settings UI can explain why a pattern behaves the way it does without changing gameplay rules.

The main scene owns local input, stage/pattern selection, stage briefing actions, Pattern Lab row selection, collision/graze checks, the compact gameplay HUD, optional debug HUD surfaces, and drawing. Future online integration should replace locally generated authoritative state with server-approved state while keeping the same presentation and debug surfaces.

## Input, Replay, And Local Authority

Local input is encoded once per simulation tick through `input_codec.gd`. The encoded state currently drives movement, focus, shooting, Bomb, and card-slot requests.

`replay_recorder.gd` records the same tick-indexed input stream plus event checkpoints such as Bomb, hit, and periodic state hashes. The recorder is intentionally lightweight: it is a local prototype surface for the replay format described in the shared docs, not a production persistence layer.

`replay_store.gd` persists the current local snapshot as JSON under `user://replays/latest_local_replay.json` and can load it back into replay playback. It also maintains `user://replays/index.json`, a small replay-list index with stable metadata such as saved time, ruleset version, seed, final tick, score, graze, hit count, pattern id, opponent code, mode code, result code, path, favorite flag, and final result hash. `replay_list_model.gd` turns that index into UI-ready row dictionaries with play/favorite/remove affordance flags, cursor state, and a selected-entry summary, while still keeping localized labels out of replay data. The main scene can refresh this model, move a local selection cursor, display the selected entry in the HUD, toggle favorite, remove an entry from the local index without deleting the replay file, and load the selected snapshot.

For now the main scene remains locally authoritative so player feel can be tuned quickly. Online play should treat this as presentation and prediction code only; Gensoulkyo will remain authoritative for accepted inputs, card events, bullet state, hits, and match results.

The local replay loop can build a snapshot from the current practice run, save/load the latest snapshot, reload the initial state, feed recorded inputs back into the fixed-tick simulation, pause/resume, change playback speed, restart playback, seek to a target tick by deterministic fast-forward, compare recorded state hash checkpoints against replayed hashes, and validate the saved final result hash when replay reaches the recorded end. Seek now has a guard against malformed snapshots that fail to advance tick state, returning a failed seek status instead of hanging the client. A mismatch is surfaced in the optional debug HUD as the first invalid tick plus expected/actual final hashes.

The gameplay HUD is compact and player-readable when practice, match, network-match, or results screens are active, showing mode, active screen, score/resources, hand/energy, current pattern, network summary, and a small menu hint. The home screen hides the gameplay HUD so the portrait-first lobby remains clean. Optional debug display still includes bullet hit circles, playfield/pickup-line guides, player hit/graze circles, recent event log lines, current input bits, replay mode, replay speed, and validation status.

Practice mode keeps a small local configuration surface: locked seed, start tick, initial power, initial Bomb count, Stage Run, and Boss Spellbook practice. Restarting practice resets the match to that configuration. Start tick is implemented by deterministic prewarm with empty input, which preserves the same pattern timeline without adding a separate seek path to the prototype simulation. Stage Run is the first local whole-stage flow: it advances through the selected stage's pattern list on a fixed phase timer, resets pattern runtime between phases, records phase-change replay events, and leaves manual pattern selection available as the focused drill mode. Boss Spellbook practice runs an authored multi-phase Boss script through the same bullet engine and visual annotation path, while staying mutually exclusive with Stage Run.

## Accessibility Prototype

`accessibility_settings.gd` owns local visual accessibility switches for the prototype:

- Low-flash mode, enabled by default.
- Simplified low-contrast background, enabled by default.
- Bullet opacity adjustment.
- Always-visible player hitbox.
- Practice graze ring visibility.
- Standard and colorblind-friendly bullet palettes.

The main scene applies these settings during drawing without changing collision rules. Bullet dictionaries retain a stable `color_name` so palette changes can be applied at presentation time instead of mutating gameplay state. The model also exposes per-option and full reset helpers so the player settings UI can restore visual defaults for low flash, background simplification, bullet alpha, hitbox visibility, graze ring, and palette choices before persistent profiles are introduced.

## Input Profile Prototype

`input_profile.gd` validates that required gameplay actions exist and have bindings: movement, focus, shoot, Bomb, card slots, and quick restart. It carries local left-hand/right-hand keyboard presets and applies them directly to Godot `InputMap`, so the HUD and smoke test have a concrete input-configuration surface before a full rebinding UI exists.

The profile model now also supports single-action rebinding, binding-row export for settings screens, restoring the current preset after a custom bind, default reset helpers for profiles, individual bindings, gamepad curve, sensitivity, deadzone, and vibration, and gamepad left-stick deadzone/sensitivity/curve mapping. The temporary settings shell can arm a capture state for a binding row, intercept the next keyboard key through `_input`, apply it to `InputMap`, show the capture status in the shell, and save the resulting binding. Keyboard input still records direction bits, while analog movement records JSON-safe `analog_x`, `analog_y`, and `analog_strength` fields so local replay snapshots can preserve gamepad movement without storing engine objects.

## Character And Shot-Type Prototype

`character_model.gd` implements the first local version of the character and shot-type docs. It defines the balanced, precision, wide-range, and spell-power bodies as config dictionaries with move speed, slow speed, normal/focus shot patterns, Bomb type and multipliers, graze modifier, spell-power modifier, and hitbox visual scale.

The main practice loop now reads movement speed, shot cadence, shot lanes, shot colors, Bomb radius/invulnerability multipliers, graze score modifier, spell-power gain modifier, and focus hitbox visual scale from the selected character. The settings screen exposes character rows with localized names and core stats. This remains local practice tuning data; future online matches should include selected character id in server-validated player snapshots so movement and shot behavior stay authoritative.

## Bullet Visual-Language Prototype

`bullet_visual_model.gd` is the first client-side implementation of the bullet visual-language docs. It reads bullet dictionaries and derives presentation-only metadata: bullet kind, speed band, danger level, visual radius, decorative alpha, outline flags, card-modified markers, homing tails, and laser-warning hints.

The main scene annotates spawned bullets and redraws active bullets from that metadata. Collision and graze checks use the live bullet `radius`, so behavior-driven radius changes such as growing-orb pulse rings, snake-stream velocity changes, and Bezier-follow position sampling affect both hit and graze overlap on the tick they are resolved. Capsule laser checks also read the live `length`, so extending lasers can warn at a short length and then grow into their full hit/graze shape. The visual model is refreshed each tick and keeps presentation radius equal to or larger than the current collision radius. Shaped bullets now include straight capsule lasers, rotating persistent capsule lasers, cross warning capsule lasers, extending capsule lasers, polyline curved lasers, reflected polyline lasers, and dynamic wave polyline lasers, all using distance-to-segment collision so hit overlap is excluded before graze overlap is counted. Dynamic wave lasers recalculate their local `points` each behavior tick, and the engine reads those live points for both hit and graze checks after the warning window expires. Laser-warning behavior is presentation-only during its warning window and becomes collidable only after the warning ticks expire, preventing warning lines from awarding graze or causing hits. Behavior bullets can also be non-collidable carriers, which lets origin-relative path emitters move scripted nodes through the playfield, lets telegraph markers show delayed burst positions, and periodically release ordinary bullets without turning the carrier itself into a hit/graze object. High-density phases reduce decorative alpha so overlapping effects do not hide the hitbox layer, and card-modified pattern bullets receive an outline marker while preserving the server-authority boundary.

## Audio Settings Prototype

`audio_settings.gd` owns the first local audio-accessibility settings model. It tracks master, music, SFX, UI, and reserved voice volume groups; exposes effective volume rows; carries switches for key-event visual cues and high-frequency graze-audio suppression; and can restore individual volume groups or all audio defaults. The main scene now creates missing Godot audio buses for those groups and applies volume/mute changes through `AudioServer`, even though production audio assets are not imported yet.

`player_settings_store.gd` persists the player-facing local settings as JSON under `user://settings/player_settings.json`. It snapshots input profile/bindings/gamepad values, audio volumes/toggles, display indices/toggles/sliders, and accessibility options, applies them on startup before runtime audio/display application, and saves automatically after supported UI setting changes, reset actions, and captured key binds. The store is intentionally local client state only; online authority and account sync remain server-owned work.

## Local I18n Prototype

`localization.gd` loads the prototype HUD and card-name text from `godot/i18n/base.en.json`, then can overlay `base.zh-CN.json` when the runtime locale changes. Gameplay code uses stable keys such as `ui.score` and `card.focus_lens.name`, while the localizer records missing keys, records required text-pack load errors, supports optional locale/theme overlays, and handles simple placeholder replacement.

This keeps visible text from spreading through gameplay code while the project is still a single-scene prototype. F12 cycles the local HUD between built-in locales so text-package behavior can be checked in-game. The loader now has the base JSON package layer and optional overlay path in place; later work can append Workshop text packages after the base locale and theme layers described by the shared i18n docs.

## Local Matchmaking And Network State

`matchmaking_model.gd` is the local client-side status and intent model for the connection flow described by the shared server docs. It creates an anonymous development session, tracks local client/config/server versions, exposes bootstrap placeholders for profile, wallet, deck, and activity pulls, and keeps the screen model supplied with login, server, config, deck, mode, network, queue, ready, and reconnect rows.

The model is deliberately not authoritative. Joining a queue only builds the same intent shape the future RPC layer will send: `session_id`, `active_deck_id`, `mode_id`, optional params, and local readiness. It validates the active deck through `deck_builder_model.gd`, blocks ranked certification when simulated ping/loss are below the documented network-quality threshold, and exposes queue/found/ready/cancel states for UI testing. Simulated match-found and reconnect helpers exist so the client can exercise status presentation before WSS match handling is wired in.

Mode rows are config-driven and currently cover certification, PvP duel, battle royale, world boss, and instance boss with mode ruleset versions, card pool ids, reward table ids, matchmaking kind, mode category, readable summaries, and the documented min/max player counts. The matchmaking page derives Quick Match, Ranked, PvP Duel, Boss Party, and Room Code cards from those client-intent rows while leaving actual match creation server-owned. Reconnect state keeps `match_id` and `session_id`, uses a 30-second local countdown, and always marks local results as non-authoritative while reconnecting or restoring.

`gensoulkyo_api_model.gd` is the first concrete client/server contract adapter for the Go server MVP. It builds HTTP request dictionaries for anonymous login, bootstrap, inventory sync, card upgrade, deck list sync, active-deck save, chest sync/open, heartbeat presence, queue join/ticket/cancel, room create/join, ready, battle allocation, signed battle ticket, battle result submit callback projection, input, snapshot, cursor event polling, mode-action submission, disconnect/reconnect, settlement, post-settlement rematch, replay read, and activity claim endpoints, including bearer-session headers and server-compatible deck snapshots. The main scene wrappers inject the current stage and character as non-authoritative `mode_params`, while `network_match_model.gd` stores the server-confirmed loadout returned by queue, heartbeat, match-start, snapshot, settlement, rematch, and replay responses. It can apply server-shaped responses back into `deck_builder_model.gd`, `chest_system.gd`, `matchmaking_model.gd`, `network_match_model.gd`, `game_mode_model.gd`, and `results_service_model.gd`, so login/bootstrap data, server-owned inventory/deck/chest projections, server card upgrades, server chest openings and audit rows, heartbeat queue/match/disconnect state, server mode configs, queue tickets, pre-match cancellation, room codes, match_start events, authoritative snapshots, event-stream cursors, server-validated mode actions, reconnect snapshots, match_end settlements, battle-result verification receipts, rematch waiting/found state, replay audit records, and server-confirmed activity claims flow through the same presentation models as the local prototype. The battle-result path is intentionally read-only from the normal client perspective: it exists to observe the C++ battle server -> Nakama/Go verification callback contract during migration, not to let the player client author rewards or match outcomes.

`gensoulkyo_http_client.gd` wraps Godot `HTTPRequest` and drives that adapter asynchronously against the local Gensoulkyo MVP. The main scene owns one instance and exposes wrappers for login/bootstrap, inventory sync, deck sync/save, chest sync/open, heartbeat, queue join, room create/join, ticket polling/cancellation, ready, battle allocation, battle ticket, battle result submit projection, input submit, snapshot, event polling, mode-action submission, disconnect/reconnect, settlement, rematch, replay read, and activity claim. The UI screen model appends prototype network actions for login/bootstrap, inventory/deck/chest sync, active deck save, server chest open, room creation, pending room-code selection, room join, ticket polling/cancellation, heartbeat, server ready, event polling, and post-match rematch, plus transport status rows so headless tests and the temporary overlay can show whether the live endpoint is idle, loading, or failed. This is enough for a local HTTP integration check; retries, offline fallback policy, WSS match streaming, and final online screen flow are still pending.

`battle_network_client_model.gd` is the first client-side boundary for the future C++ battle transport. It syncs battle allocation and signed ticket data from `network_match_model.gd`, validates descriptor readiness through `protocol_descriptor_model.gd`, models the intended KCP/UDP transport, X25519 ECDHE + HKDF handshake target, protobuf phk.v1 codec, ChaCha20-Poly1305 pending state, and seq/ack/nonce packet header shape. The Network Match page now exposes Prepare Handshake, Connect Scaffold, and Build Input Packet actions that drive this model through the same card/quick-action/row pipeline as other menu controls. These actions are deterministic migration scaffolds for tests and UI integration; they do not send UDP traffic, encrypt payloads, or replace the pending GDExtension/native battle client.

The current Gensoulkyo MVP snapshot now includes deterministic server-side bullet deltas, server-approved active cards, and mode-state Boss HP/damage projections. SpellKard still treats these as authoritative network presentation data through `network_match_model.gd`; the local practice simulator remains separate tuning code until the shared bullet/card rules are extracted into a common server/client ruleset.

## Server-Authoritative Network Match Surface

`network_match_model.gd` is the first client-side bridge for the authoritative match docs. It does not replace the local practice loop yet; instead it models the online surfaces that a WSS match adapter will drive: loading, ready, running, ended, tick-indexed input packets, server-confirmed loadouts, server snapshots, server bullet projections, server active-card projections, server event logs, correction metrics, full-snapshot requests, mode-state summaries, battle ticket state, battle-result verification/audit state, and authoritative online replay metadata.

Outgoing input packets contain only the allowed client fields: tick, seq, direction bitmask, slow, shoot, Bomb, and card slot. The model rejects non-monotonic ticks, invalid seq values, invalid direction bits, out-of-range card slots, and forbidden client-authored result fields such as score, graze, damage, rewards, Boss HP, or rank score. Input delay is kept in the documented 2-4 tick range and can be adjusted from simulated ping/loss/jitter so latency smoke tests have a concrete client metric.

Incoming snapshots are treated as server authority. The model accepts full and delta snapshots, tracks state hashes, merges mode display state, applies `bullets_delta` operations into a server-bullet table, refreshes the server active-card table from `active_cards`, records recent server events, classifies predicted-vs-server position error as smooth, interpolate, resimulate, or hard snap, records correction count, hard snap count, average position error, late input count, perceived hit mismatch, and resimulated ticks, and requests a full snapshot when a hard correction or hash mismatch requires it. Full snapshots rebuild the server-bullet table from `sync` deltas; delta snapshots apply `spawn`, `move`, and `despawn` operations. The model can also consume Gensoulkyo's cursor-based event stream and heartbeat presence, tracking cursor/latest/has-more plus match tick/connected/reconnect status while appending authoritative server events into the same recent-event presentation log. While a network match is running, the main draw path can render the projected server bullets as presentation-only objects without feeding them into local practice collisions or scoring. `match_end` events are the source of online replay ids and final results, preserving the rule that the client cannot derive rewards or outcomes locally. The battle-result submit response projection records accepted/rejected status, settlement key, duplicate flag, result hash, replay id, and key id only after a server-authoritative receipt; match-id mismatches are rejected locally as contract failures. After settlement, server rematch responses are treated as authority too: waiting state is displayed, and a found response resets the network surface into a fresh loading match using the server-provided new match id, mode, stage, and loadout before the normal ready/start path resumes. The client can then read the server-generated replay audit record for the original replay id and stores its state hash, input count, and event count as metadata only. When `match_end` carries server reward, task, event, leaderboard, or mode-result fields, the main scene converts it into the same server-shaped result object accepted by `results_service_model.gd`; replay-only `match_end` events remain network metadata only.

## Local Game Mode State Surface

`game_mode_model.gd` turns the four game-mode docs into UI-ready client state while preserving the shared server-authority boundary. Certification rows show rating code, rank score, top-30% unlock progress, challenge stage id, and pass-condition keys. The client can apply a server result for display, but does not compute rating, rank deltas, percentile, or next-stage qualification itself.

Battle royale rows track 5-10 player readiness, the server/shared match card pool shape, fixed 30-second rounds, three server-issued candidate cards, selected-card requests, zero-round order, and effect trigger tick. The mode UI decorates the candidate row with the next selectable card so Enter can submit a `select_round_card` request. The model validates that each player contributes four cards to the shared pool and that selection requests reference one of the three candidates, but the candidates, effect timing, ranks, and rewards remain server-owned.

World Boss and instance Boss rows share party-position and card-transfer surfaces. The mode UI derives a default sender, receiver, and untransferred active-deck card so Enter can submit a `transfer_card` request. The model validates 4-8 party size, computes display-only spawn/aim directions around a center Boss, records one card-transfer request per card, and rejects duplicate transfer attempts. World Boss state displays persistent HP, daily attempts, and one-time defeat notice values received from the server. Instance Boss state displays phase, clear conditions, clear/fail exclusivity, and star count from server-like result data without treating local damage or Boss HP as authoritative.

## Local Results, Rewards, And Activity Surface

`results_service_model.gd` is the first client-facing result and reward presentation layer for the server economy docs. It accepts server-shaped match settlements with `match_id`, `user_id`, `reward_json`, score/stat summaries, task progress, event points, leaderboard updates, mode result data, and `replay_id`, then turns them into result-screen and activity-center rows. It updates a local display wallet and local open-source inventory surfaces only from those server-shaped settlements.

The model enforces the important economy boundary: rewards are idempotent by `match_id:user_id`, duplicate result submissions return the already stored result without granting again, compensation claims are idempotent by `compensation_id`, activity claim confirmations are idempotent by `claim_kind:claim_id:user_id`, and results marked as client-authored rewards are rejected. Reward ledger rows preserve source, amount, item id, match id, user id, and creation time so later audit UI can show why a wallet/card/chest changed. Tasks, events, and leaderboards are display projections of server updates. Their activity rows can produce `pending_server` claim requests with `client_result_authoritative=false`, but those requests do not mutate wallet, inventory, task-claimed flags, event rewards, or leaderboard rewards locally. Only server-shaped activity claim confirmations can apply `reward_json`, mark tasks claimed, or move event/leaderboard reward state to claimed.

## Local Chest Prototype

`chest_system.gd` implements the open-source local chest surface from the shared economy docs. It owns local wallet values, owned chest counts, enabled pool configs, weighted rarity rows, 10-open rare pity, 60-open epic pity, result rows, and audit records. Opening a chest deducts local cost, uses a deterministic local seed for the prototype roll, grants the resulting card through `deck_builder_model.gd`, converts overflow copies to card dust, and records cost/seed/result/time in `user://chests/local_chests.json`.

For online play the same model is now also a projection target for Gensoulkyo chest snapshots and chest-open responses. Server responses replace local wallet/chest/pool/pity state, append server audit records, display returned results, and refresh the deck-builder inventory from the server-owned inventory snapshot. The client still sends only `pool_id` and `count`; roll results, cost deduction, duplicate conversion, pity updates, and audit ids remain server-owned.

This is intentionally not the Steam Inventory or market implementation. Commercial ownership, itemdef, market attributes, fraud controls, and server-issued drops remain outside the open-source client layer. The local model exists so the UI, inventory, deck validation, and result display flows can be developed and tested without closed services.

## Asset And Theme Manifests

`godot/assets/asset_manifest.json` is the machine-readable source of truth for imported media. The current prototype has no production media records yet, which is intentional: placeholder directories and README files are allowed, but future sprites, audio, raw source files, and theme media must be listed there with an allowed license before they are committed.

`godot/themes/base/theme_manifest.json` defines the original base theme package shape and points its asset license field back to the asset manifest. Theme packages may replace presentation resources such as text, sprites, audio, and UI skinning, but they must not change gameplay rules, hit radii, card numbers, or replay data.

`theme_registry.gd` is the runtime bridge for theme manifests. The main scene loads the base theme through this registry, exposes the active theme/version/replacement scope in the HUD, and passes the active theme id into `localization.gd` so theme text overlays can be layered without changing gameplay code. The registry can scan a local Workshop-style directory under `user://workshop/themes`, registering valid theme manifests while rejecting packages that attempt to replace gameplay rules or other forbidden surfaces. Activating a discovered theme reloads base locale text and appends the active theme's declared text packs for the current locale, while replay/gameplay state continues to store only stable codes and keys.

## Local Card Prototype

`deck_builder_model.gd` provides the local deck-construction surface described by the shared UI and deck-rule docs. It owns a prototype catalog with stable card ids, i18n name keys, rarity, type, cost, ruleset version, ranked-ban flags, and card effects. It exposes UI-ready card rows with rarity/type filters and add/remove affordance metadata, tracks dirty edit status, lets the UI modify the working 20-card deck, validates that deck against ownership, max-two-copy, high-rarity, strong-interference, ranked-ban, and ruleset constraints, rejects invalid saves, saves valid local decks under `user://decks/local_decks.json`, marks one deck active, and exports a stable `deck_snapshot` with only codes and card ids.

For online play it also projects server inventory levels and applies server-authoritative card-upgrade responses. Local upgrade-cost preview rows mirror the open-source rarity/dust ladder for UI feedback, but wallet deduction, target-level validation, max-level enforcement, and the final inventory level are accepted only from Gensoulkyo.

`card_system.gd` consumes the active deck definitions from the builder for local practice and consumes saved replay deck snapshots during replay playback. This keeps combat behavior tied to the same deck data the UI validates while preserving the replay rule that localized text and theme assets are not stored in replay packages.

The local SpellCard runtime currently provides:

- A practice deck, four-card hand limit, fixed draw interval, energy costs, and cooldowns.
- Self modifiers for focus speed, hitbox shrink, Bomb radius/invulnerability, graze value, and shield charges.
- Pattern modifiers for speed, density, angle, aim bias, graze value, and score multiplier pressure.
- Tick-indexed card accepted/rejected events for replay inspection.
- Basic resource economy effects such as energy gain and draw-card effects.

Cards expose modifiers rather than directly editing live bullet arrays. This mirrors the shared design boundary: cards affect generation parameters, player survivability, or scoring state, while authoritative bullet and hit outcomes remain produced by the simulation layer.
